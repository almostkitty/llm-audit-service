# llm-audit-service

Сервис аудита использования генеративных языковых моделей для профессиональных текстов.

## Статус проекта

MVP дипломного проекта: анализ текста по эвристическим метрикам, детекция скрытых Unicode-сигналов, простая страница для проверки API и отдельная страница для извлечения текста из документов.

## Что уже реализовано

- FastAPI, точка входа `app/main.py`
- **`POST /audit`** — загрузка **текстового** файла (UTF-8), возврат метрик, `hidden_unicode` и `llm_probability`
- **`POST /extract`** — загрузка **PDF, DOCX, RTF, ODT или .txt**; ответ `{"text": "..."}`
- **`GET /demo`** — `templates/demo.html`: вставка текста → запрос к `/audit`, краткая сводка по Unicode-сигналам и полный JSON
- **`GET /extract`** — `templates/extract.html` (файл → запрос к `/extract`)
- Препроцессинг: lower-case, нормализация пробелов (`preprocessing/cleaner.py`); он применяется **после** подсчёта `hidden_unicode`, чтобы не «съедать» неразрывные и узкие пробелы
- Числовые метрики (см. раздел внизу) считаются по тексту **после** `clean_text` и попадают в `metrics` ответа **`/audit`**
- **`hidden_unicode`**: эвристика по невидимым символам, bidi-маркерам и литералам `\$` / `\~` — `metrics/hidden_unicode.py`, считается по **исходной** строке (типографские тире и «умные» кавычки в коде **отключены**, их можно снова включить в том же файле при необходимости)
- **`llm_probability`**: эвристика в `scoring/aggregator.py` (в скор входят только четыре метрики из блока `metrics` — см. ниже); `hidden_unicode` в агрегатор **не входит**
- Извлечение текста:
  - **PDF** — `pymupdf4llm` (`pdf_reader.py`)
  - **DOCX** — `python-docx` (`docx_reader.py`)
  - **RTF** — `striprtf` (`rtf_reader.py`)
  - **ODT** — `odfpy` (`odt_reader.py`)

## Текущие ограничения MVP

- **`/audit`** принимает только содержимое как **UTF-8 текст** в поле `file`;
- Бинарный **`.doc`** (Word 97–2003) **не поддерживается**;
- Роуты `auth`, `files`, `reports` — заготовки;
- Скоринг без ML и без калибровки на датасете; веса в `aggregator.py` — baseline

## API: POST /audit

`multipart/form-data`, поле `file` — **текст в UTF-8** (например `.txt`).

Пример:

```bash
curl -X POST "http://127.0.0.1:8000/audit" \
  -H "accept: application/json" \
  -F "file=@sample.txt"
```

Остальные поля в `metrics`, а также объект `hidden_unicode`, в сумму **не входят** — они для анализа и прозрачности ответа.

## API: POST /extract

`multipart/form-data`, поле `file`:

| Расширение | Обработка |
|------------|-----------|
| **`.pdf`** | pymupdf4llm |
| **`.docx`** | python-docx |
| **`.rtf`** | пакет `striprtf` |
| **`.odt`** | пакет `odfpy` |
| **`.txt`** / без расширения | UTF-8 (битые байты — с заменой) |

Ответ: `{"text": "извлечённый текст"}`.

## Как интерпретировать результат анализа

- **`llm_probability`** — эвристический скор по четырём метрикам: ближе к `0` — ниже условная «оценка LLM-паттерна», ближе к `1` — выше (значение ограничено сверху `1.0`).
- **`hidden_unicode`** — отдельный сигнал: наличие подозрительных символов и экранирований; **не смешивается** с `llm_probability`. Интерпретировать в связке с источником текста (копипаст из чат-бота vs чистый набор в редакторе).

## Экспериментальный корпус (пилот, n = 12)

Зафиксировано для сопоставимого сравнения **human** vs **LLM** на одном наборе тем (ВКР студентов НИУ ВШЭ, ФКН, период подготовки работ — см. исходные публикации; тексты human подготовлены вручную: без титула, оглавления, списка литературы и т.п.).

### Human

- Карточки тем и метаданные: `data/works/manual_2018/text-data.txt` (поля `#file`, `#title`, `#abstract-ru`, `#chapters`, `#human-words`).
- Исходные выдержки для метрик: `data/works/manual_2018/text1.txt` … `text12.txt` (ручная очистка).

### LLM (DeepSeek)

- **Модель:** `deepseek-chat` (DeepSeek API).
- **Параметры генерации (базовый прогон):** `temperature = 1.0` (как рекомендует документация DeepSeek для обобщённых задач; для корпуса фиксируется в протоколе эксперимента), `max_tokens` задаётся при запуске CLI (длинные работы — увеличивать, чтобы не обрезало вывод).
- **Промпт:** скрипт генерирует **содержательный текст глав 1–3**, без реферата, abstract, введения, заключения и списка источников; целевой объём — около `#human-words` с допуском ±10% (инструкция модели, фактический объём зависит от ответа API).
- **Код запуска:** `extra/llm-texts.py` (`--mode batch-from-text-data`, вывод в каталог по умолчанию `data/works/manual_2018/llm/`, опционально `--json-out-dir` для сырых ответов API).

### Таблицы метрик

| Файл | Содержание |
|------|------------|
| `extra/manifest.csv` | метрики по **human** (те же колонки, что у `analyze_text`) |
| `extra/manifest_llm.csv` | метрики по **LLM** |
| `extra/manifest_summary.csv` | описательная статистика по human (`describe`) |
| `extra/manifest_llm_summary.csv` | описательная статистика по LLM |

Сборка манифеста по папке с `.txt` (берутся только корпусные файлы `textN.txt`):

```bash
export ENABLE_PERPLEXITY=1
python3 scripts/build_manifest_from_dir.py data/works/manual_2018/llm extra/manifest_llm.csv
```

Для Excel в русской локали можно вывести с `;`:  
`python3 scripts/build_manifest_from_dir.py ... extra/manifest_llm.csv --delimiter ';'`

Описательная статистика по любому manifest:

```bash
python3 scripts/describe_manifest.py extra/manifest.csv --out extra/manifest_summary.csv
python3 scripts/describe_manifest.py extra/manifest_llm.csv --out extra/manifest_llm_summary.csv
```

**Заметка по `unicode`:** колонка отражает `hidden_unicode.has_signals`. На пилоте LLM-тексты могут давать **0** при тех же правилах — это не «доказательство отсутствия артефактов у всех LLM**, а результат конкретного пайплайна и корпуса.

## Roadmap перплексия, корпус и оценка качества


### Исходная точка

- Есть **~300 текстов**, заведомо **human-generated** — это уже полноценный **якорный корпус** для описательной статистики и для сравнения с машинными вариантами (не обязательно для «обучения с нуля», но очень полезно для **эксперимента** и для защиты работы).
- Цель этапа: (1) зафиксировать **протокол** данных, (2) получить **сопоставимый** набор LLM-текстов, (3) реализовать **перплексию** на одной выбранной языковой модели, (4) прогнать корпуса и **интерпретировать** результаты в дипломе (с оговорками из теоретической части).

### Этап 1. Корпус human и воспроизводимость

1. **Инвентаризация:** список файлов (id, путь, длина в символах/токенах, кодировка). Исключить пустые и явно битые извлечения.
2. **Единое извлечение текста:** для PDF/DOCX/ODT/RTF — тот же пайплайн, что в сервисе (`ingestion/`), результат — нормализованный `.txt` или JSONL со полем `text` (чтобы пересчёты были воспроизводимыми).
3. **Минимальная документация для ВКР:** откуда тексты (источник/жанр), критерий «точно человек», дата скачивания. Этого достаточно для раздела «Данные» без претензии на репрезентативность всего интернета.

### Этап 2. Параллельный корпус LLM (рекомендуемая схема)

Чтобы сравнение было честным, лучше не «случайные промпты», а **связка с human-текстами**:

| Подход | Суть | Плюсы / минусы |
|--------|------|----------------|
| **Перефраз / продолжение по отрывку** | Берётся заголовок, первый абзац или краткое резюме human-текста → LLM генерирует **новый** текст в том же жанре | Жанр ближе; всё равно возможен сдвиг стиля |
| **Парный датасет** | На каждый human-файл — один (или несколько) LLM-ответов при **фиксированном** промпте и версии модели | Удобно считать разницу метрик и строить таблицы для ВКР |
| **Несколько температур** | Одна модель, разные `temperature` / top-p | Показывает чувствительность PPL и прочих метрик к настройкам генерации (хорошо для «ограничений методов») |

Обязательно **зафиксировать в логе:** модель (имя, версия API или чекпоинт), параметры генерации, дата прогона, хэш промпта. Без этого в дипломе нельзя воспроизвести эксперимент.

Объём **~300 пар** уже достаточен для **описательной** статистики и простых разделений (например, медиана PPL human vs LLM, пересечение распределений). Для уверенных заявлений про ROC/AUC позже может понадобиться больше данных или кросс-валидация — это можно честно описать как ограничение.

### Этап 3. Реализация перплексии (техническая часть)

1. **Выбор одной каузальной LM** с приемлемым русским языком (многоязычная или русскоязычная; одна модель на весь эксперимент). Обосновать выбор в ВКР: размер, лицензия, доступность на твоём железе (CPU допустим для маленьких моделей; GPU ускоряет).
2. **Реализация в `metrics/perplexity.py`:** токенизация → forward с учётом длины контекста → усреднение NLL по токенам → `ppl = exp(mean_nll)`; для длинных текстов — **скользящее окно / чанки** с явной формулой усреднения (описать в тексте один раз).
3. **Согласование с препроцессингом:** решить, считать PPL по **сырому** тексту или по лёгкой нормализации, и **не** смешивать с агрессивным `clean_text`, если одновременно важны Unicode-сигналы (для диплома достаточно явно назвать две ветки: «метрики после cleaner» vs «PPL по исходнику»).

### Этап 4. Прогон по корпусам и анализ (что писать в «Практика»)

1. Batch-скрипт (отдельно от FastAPI): читает каталог human и каталог llm, пишет CSV/JSONL: `id`, `ppl`, при желании остальные метрики из `analyzer`.
2. **Описательная статистика:** медиана, квартили, доля пересечения диапазонов human/LLM — честно показать, что классы могут сильно пересекаться (как у тебя в теории).
3. **Опционально:** простой порог по PPL, ROC на парных данных, или логистическая регрессия по **вектору признаков** (включая PPL) — как шаг к roadmap ниже; не обязателен для минимально сильной работы, если хорошо описан эксперимент и ограничения.

### Этап 5. Финализация ВКР и калибровка

1. При необходимости включить PPL в `analyzer` через `ENABLE_PERPLEXITY=1` и зафиксировать режим запуска в протоколе эксперимента.
2. Обновить **`scoring/aggregator.py`** только после того, как на твоих данных будет ясно, имеет ли PPL вес и какой знак (не делать «магические» веса без обоснования).
3. В тексте ВКР: формула перплексии, выбор модели, протокол корпуса, таблицы/графики распределений, выводы и ограничения (жанр, перефразирование человеком, смена модели генерации).

### Прочий roadmap продукта (вне фокуса перплексии)

- Подключить **`/audit`** к загрузке бинарных документов (PDF/DOCX) без ручного копирования текста
- Отчёты, хранение результатов аудита, авторизация
- При наличии размеченных данных — **калибровка** `llm_probability` (нормализация признаков, логрег или иной агрегатор)
